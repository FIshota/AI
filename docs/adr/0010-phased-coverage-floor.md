# ADR 0010: テストカバレッジ CI gate は段階制 (Phase 2: 60% / Phase 3: 80%)

- Status: Accepted
- Date: 2026-04-23
- Deciders: honnsipittu

## コンテキスト

共通規約 (common/testing.md) は「最低 80% カバレッジ」を定めている。
一方、ai-chan は 157 ファイル / 57,962 行の単独開発プロダクトであり、
Phase 2 時点で一部モジュール (UI、skills プラグイン境界、launchd 連携) は
まだテストが薄い。初日から一律 80% を強制すると以下が起きた:

- 機能追加のたびに CI red、イテレーション停止
- 形式的な「行カバレッジ稼ぎ」テストが増える
- 価値の高いテスト (消去権ドリル、HinoMotoBridge 契約テスト) に回すべき
  時間が奪われる

しかし何もしなければカバレッジは退行する。

## 決定

カバレッジ CI ゲートを **段階制** にする。

| Phase | 床 (floor) | 退行検出 |
|---|---|---|
| Phase 2 (現在) | **60%** 以上 | 直前コミットより 1% 以上の低下で warn |
| Phase 3 昇格時 | **80%** 以上 | 共通規約準拠 |
| Phase 4+ | 80% 維持 + 重要モジュール 90% | 個別指定 |

Phase 3 昇格条件 (docs/ROADMAP.md) に「カバレッジ 80% 達成」を明記する。
それまでに薄い領域 (UI / skills 境界) のテスト強化を計画に入れる。

## 理由

- **実装速度と品質のバランス**: 単独開発では漸進的引き上げのほうが
  最終到達点が高い (経験則)。
- **退行の防止**: 床を明示すれば無秩序な低下は起きない。
  1% 退行 warn で意図しない劣化を早期検知。
- **重要モジュールの優先**: 消去権 (ADR 0006)、HinoMotoBridge (ADR 0005)、
  コーパス境界 (ADR 0002) は Phase 2 時点でも 80% 以上を目標として個別管理。
- **共通規約との整合**: common/testing.md は「言語規約で上書き可」と規定
  しており、本 ADR はプロジェクト段階事情による override として位置付ける。

## 結果 / トレードオフ

- CI 設定 (`.github/workflows/` もしくは Makefile `test` ターゲット) に
  `--cov-fail-under=60` を設定。
- Phase 3 昇格時に値を `80` に更新する PR を予約。
- トレードオフ: 60% は共通規約の妥協。重要モジュール個別 80% の運用で
  補う。
- 退行検知 (1% 以上の低下で warn) は履歴比較が必要なため、
  coverage.xml を artifact に保存し前回と diff する運用を入れる。

## 代替案 (検討して却下)

### 案 A: 最初から 80% を強制
却下理由: イテレーション停止、形式的テスト増加。本プロジェクトの段階で
害のほうが大きい。

### 案 B: カバレッジ gate を持たない (warn のみ)
却下理由: 単独開発で退行が不可避。明示床がないと徐々に下がる。

### 案 C: 行ではなく branch coverage を採用
却下理由: 追加の価値はあるが、ツール設定 / 読解コスト増。
Phase 3 以降の選択肢として保留。
