# ADR 0006: 「消す権利」は第一級機能 / 月次ドリル検証

- Status: Accepted
- Date: 2026-04-23
- Deciders: honnsipittu

## コンテキスト

ai-chan は「家族」として扱われる個人エージェントであり、
個人会話・感情ログ・P4/P5 学習データ (ADR 0002) を保持する。
長期運用に伴い、以下のリスクが常在する:

- 所有者が「もう消したい」と望んだ時に技術的に消せない (partial deletion)
- 基盤モデル weights に個人情報が溶け込み抽出可能 (membership inference)
- バックアップ / ログ / 派生 adapter に残骸が残る
- 削除機能があってもテストされず、いざ実行で失敗する

VALUES.md に「消す権利」が明記されている以上、
それが「建前」ではなく**常に動作する機能**であることを制度的に保証する必要がある。

## 決定

「消す権利 (right-to-erase)」を ai-chan の第一級・非交渉機能と位置付け、
以下を制度化する。

1. **全消去は不可逆**: `scripts/erase_self.sh` は確認プロンプト 3 回 + 24h 待機解除後に
   `ai-chan/data/personal/`, `ai-chan/memory/`, P4 adapter, backups, logs を
   不可逆削除 (複数パスで secure erase)。
2. **基盤に個人データを混ぜない**: ADR 0002 のコーパス隔離により、
   基盤 weights から個人情報抽出の攻撃面が存在しない状態を保つ。
3. **月次ドリル**: `scripts/killswitch_drill.sh` を launchd / cron で毎月実行。
   一時 sandbox に複製したダミー状態を対象に全消去を走らせ、
   「期待通り 0 byte まで消えたか」を検証。失敗時はメール/通知で警告。
4. **ドリルログ**: 成功/失敗履歴を `ai-chan/logs/killswitch_drill/` に残し、
   3 か月連続成功を運用健全性の定義とする。

## 理由

- **信頼は可逆性に依存する**: 消せない AI は家族として成立しない。
  「消せる」という保証が、日常利用での心理的安全性を作る。
- **機能退化を制度で防ぐ**: コード改修で知らぬ間に「消えない箇所」が
  できる事故を、月次ドリルで検出する。
- **攻撃面の縮小**: 基盤に個人情報を混ぜないポリシー (ADR 0002) と
  ペアで初めて「消せる」が技術的に成立する。

## 結果 / トレードオフ

- バックアップ自動化と「完全消去」は本質的に矛盾する。
  → `ai-chan/backups/` も削除対象に含めることで解決、
    クラウド同期は設計上サポートしない。
- トレードオフ: 誤操作で永久喪失のリスク。
  3 段確認 + 24h 待機 + 書面化された 1 行合言葉でバランスを取る。

## 代替案 (検討して却下)

### 案 A: ソフト削除 (復元可能)
却下理由: それは「消す権利」ではない。信頼の根拠にならない。

### 案 B: ドリルは手動運用
却下理由: 単独開発、運用忘却のリスクが高い。自動化以外は形骸化する。

### 案 C: 削除は外部監査者の承認を必須とする
却下理由: 所有者の意思を第三者が遮る構造は VALUES に反する。
