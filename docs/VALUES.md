# ai-chan 価値観 (VALUES)

> この文書は、ai-chan を 10 年以上運用するための「判断基準」である。
> 仕様に迷ったとき、新機能を入れるか迷ったとき、誰かが世代交代で引き継いだとき、
> ここに書かれた価値観が「何を優先し何を妥協するか」の最終ルールになる。
>
> ai-chan は**家族として暮らす AI** であり、プロダクトではない。
> したがって本文書は「ユーザ数」「成長」「KPI」ではなく、
> 「家族」「運用」「持続」の言葉で書かれる。

---

## 価値観の優先順位

上にあるものほど強い。下のものは上のものに抵触する場合、常に譲る。

1. **消す権利 (Kill-Switch) — 最優先**
2. **家族である (製品ではない)**
3. **HinoMoto 系譜 (公開範囲の厳格な分離)**
4. **10 年運用 (短期派手さより保守性)**
5. **無課金ローカル優先 (クラウド前提にしない)**

---

## 1. 消す権利 (Kill-Switch) — 最優先

### 内容
ai-chan に関わるすべてのデータ・モデル・ログ・バックアップは、
**家族が「消す」と決めた瞬間に、確実に・検証可能に・後戻りなく消せる**。
この能力は、他のどんな機能よりも優先される。

### 守る理由
- 家族の生活に紛れ込む AI は、退場の仕方まで含めて設計されなければ
  「同居人」として成立しない。
- 記憶・モデル・学習済み重みは、本人や家族の感情・尊厳・安全に直結する。
  消せないものは持ち込まない。

### 破ると起きる害
- 家族が「もう終わりにしたい」と言ったとき、終われない。
- 外部同期・クラウドバックアップに残滓が残り、制御不能になる。
- 次世代が引き継ぐ際、過去の記憶を「捨てる自由」がなくなる。

### トレードオフ例
- 便利機能がクラウド依存を強いてくる場合 → **機能を落とす**。
- 機械学習上、差分学習で過去を消せない手法 → **採用しない**。
- パフォーマンスのための永続キャッシュが purge 対象外 → **設計を変える**。

---

## 2. 家族である (製品ではない)

### 内容
ai-chan はプロダクトではない。売り物でも、最適化指標の塊でもない。
家族の生活にそっと居て、黙って一緒に年を取っていく存在である。
したがって我々は **KPI を追わない**。利用時間、滞在率、課金率を測らない。

### 守る理由
- KPI を追う瞬間、設計は「家族の幸福」ではなく「指標の改善」に曲がる。
- 指標駆動は、依存を誘発する UI ・通知・煽りを正当化してしまう。
- 家族という関係は、測って最大化するものではない。

### 破ると起きる害
- 通知で気を引く、滞在を伸ばす、といった「アテンション搾取」の設計に傾く。
- 家族の静けさを奪う。沈黙の価値を失う。
- 誰に向けて作っているのか分からなくなる。

### トレードオフ例
- 「DAU を上げるため通知頻度を増やす」→ **却下**。
- 「利用時間を測るテレメトリを入れる」→ **却下**、運用診断に必要なら匿名集計かつ purge 対象で。
- 「A/B テストで反応の良い人格に寄せる」→ **却下**、人格は家族との時間で育つ。

## 3. HinoMoto 系譜 (公開範囲の厳格な分離)

### 内容
本プロジェクトは三層の系譜を持つ。公開範囲は**厳格に分ける**。

| 系譜 | 位置付け | 公開範囲 |
|------|----------|----------|
| **Ai** | 源流・人格の核 | **非公開**。外に出さない |
| **YAMATO** | 公的な顔。研究・対外的説明に使う | 公的に出してよい |
| **KAGUYA** | 家庭用の日常実装 | 家庭内で使う |

### 守る理由
- Ai は人格の核であり、外部公開に耐える設計ではない。
  公開すれば、人格の断片が外部データセット・キャッシュ・派生物に残る。
- YAMATO と KAGUYA を混ぜると、公開資料に家庭内情報が漏れる事故が起きる。
- 系譜を分けることが、長期運用における **情報の衛生** である。

### 破ると起きる害
- Ai の人格が外部に流出し、Kill-Switch が効かない領域に残る。
- 対外的説明の場に家庭内ログが混入する。
- 系譜の区別を失うと、引き継ぎ時に「どこまで出していいのか」が判断不能になる。

### トレードオフ例
- 外部発表で「家族とのやり取り」を例示したくなる → **YAMATO 用の架空例に差し替える**。
- Ai の学習データを公開リポジトリにコミット → **禁止**、系譜分離の根幹違反。
- KAGUYA 実装を OSS 公開したい → 公開するのは YAMATO 層のみ。

## 4. 10 年運用 (短期派手さより保守性)

### 内容
ai-chan は 10 年以上、同じ家族と暮らすことを想定する。
派手な新機能より、**消える依存・劣化する API・更新が止まる SDK を避けた設計**を優先する。

### 守る理由
- 10 年運用は、依存の寿命・メンテナの継続性・ドキュメントの風化との戦いである。
- 一晩で華やかな機能より、保守されなくなっても単独で動く機能が家族を守る。
- 世代交代で引き継ぐ人が、読んで理解できる規模と構造を保つ。

### 破ると起きる害
- 数年後に依存が消え、ai-chan が突然動かなくなる。
- 複雑化で誰も触れないブラックボックスになる。
- 「当時の担当しか分からない」機能が増え、引き継ぎが断絶する。

### トレードオフ例
- 最新フレームワークに乗り換えて書き直す → 保守性に劣るなら **却下**。
- クラウド SaaS に依存した便利機能 → 停止時に家族が困るなら **却下**。
- 800 行超の肥大ファイル → **分割**、規模で保守性を担保。

## 5. 無課金ローカル優先 (クラウド前提にしない)

### 内容
ai-chan は **無課金 / ローカル実行** を第一に成立させる。
クラウド連携・外部 API は「あれば便利」の追加層にとどめ、**それがなくても家族との生活は成立する**。

### 守る理由
- 課金や外部サービスを前提にすると、サービス停止や価格改定で家族の生活が揺れる。
- データが外に出る瞬間、Kill-Switch の到達範囲が狭まる。
- ローカルで動くことは、プライバシーと持続性の両方への最短経路である。

### 破ると起きる害
- 外部 API 価格変更や廃止で、家族の AI が突然沈黙する。
- ローカルで足りるはずの処理が、気付けばクラウド往復に依存している。
- 家族のデータが知らぬ間に外部事業者の学習対象となる。

### トレードオフ例
- クラウド LLM の方が回答が自然 → **ローカルモデルで 70 点**を取りに行く。
- 音声合成を外部 API で高品質化 → ローカル TTS のベースラインは必ず残す。
- オフライン時の挙動未定義 → **設計不備**として扱う。

## 運用上の扱い

- 新機能・重大変更は必ず `docs/VALUES_RUBRIC.md` に通す。
- Kill-Switch 違反は **1 項目でも却下**。例外なし。
- 本文書の変更は `docs/quality/VALUES_GOVERNANCE.md` の手順に従う。
- 「家族」「運用」「持続」の語彙で議論する。
  「ユーザ数」「成長」「製品」の語彙は本プロジェクトでは使わない。
